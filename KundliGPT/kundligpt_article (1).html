The chart is the moat, not the model

How to build a chart-grounded Vedic astrology assistant that reads sharp instead of generic — by computing the kundli yourself and keeping the language model pluggable.

Swiss Ephemeris FastAPI LLM grounding Claude / Groq ~12 min read

If you wire a language model to a third-party astrology API and the answers still feel like newspaper horoscopes, the model isn't the problem. The grounding is. This is a build log for a KundliGPT-style assistant that fixes the grounding first — and treats the model as the last, swappable step.

The premise is simple to state and easy to get wrong. A good astrology chatbot has to do two very different jobs: compute an accurate birth chart, and interpret it well. Most weak implementations collapse those jobs into one — they ask an API for a pre-written "Saturn in the 7th house means…" blurb and hand it to the model to reword. The model has nothing specific to reason about, so it drifts toward the generic. The fix is to separate the two jobs cleanly, own the first one entirely, and give the model exact facts to interpret rather than prose to paraphrase.

The accuracy a user feels comes almost entirely from the deterministic layer. The model only ever interprets what that layer computes.

01Grounding beats paraphrase

The single most important decision is to compute the chart yourself with Swiss Ephemeris instead of renting interpreted text from an astrology API. Swiss Ephemeris gives you raw planetary longitudes to arc-second precision for any moment and place. From those raw positions you derive everything downstream: the ascendant, house cusps, sidereal sign placements, nakshatras and padas, retrogrades, the Vimshottari dasha timeline, and current transits.

That ownership is what makes the model's answers specific. Instead of "your Saturn placement suggests patience," the model is handed Saturn at 20.33° Scorpio, house 2, Jyeshtha nakshatra, currently in a Rahu–Moon dasha period ending December 2027 — and asked only to interpret. Interpretation over exact data is a far more constrained task than generating astrology from a vague prompt, and constrained tasks are where models are reliable.

02The shape of the system

Three layers, read top to bottom along the request path, with the external services each layer talks to on the right. The compute layer in the middle is where the real work lives.

03The five layers

Step 1 — Compute the chart (the moat)

Set the sidereal mode to Lahiri, convert the birth moment to Julian Day in UT, then read each body's sidereal longitude. Whole-sign houses are counted from the ascendant's sign. Everything else — nakshatra, pada, dignity — falls out of the longitude.

import swisseph as swe

swe.set_sid_mode(swe.SIDM_LAHIRI, 0, 0)
jd    = swe.julday(y, m, d, hour_ut, swe.GREG_CAL)
flags = swe.FLG_SWIEPH | swe.FLG_SIDEREAL | swe.FLG_SPEED

lon, speed = swe.calc_ut(jd, swe.SATURN, flags)[0][0], _
sign  = SIGNS[int(lon // 30)]           # sidereal sign
house = ((int(lon//30) - asc_sign_idx) % 12) + 1  # whole-sign
astro/ephemeris.py — verified: Lahiri ayanamsa 23.857° for 2000; Gandhi's chart → Libra ascendant.

Step 2 — Serialize a dense context block

Turn the chart into exact, machine-derived facts the model reasons over — not prose to reword. This block is the difference between sharp and generic.

Step 3 — Interpret, don't calculate

The system prompt gives the model a persona and a rule scaffold, and — crucially — tells it the numbers are already correct. Its job is to reason from house lords and the running dasha, cite the logic in plain language, and stay honest about astrology being guidance rather than fate.

Step 4 — The timing engine

Vimshottari dasha is what produces the "when does this period begin or end" answers people find uncanny. It's derived from the Moon's nakshatra at birth: a 120-year cycle of nine planetary lords, with the balance of the first period set by how far the Moon has moved through its nakshatra.

# balance of the birth mahadasha
frac_elapsed  = (moon_lon % NAK_SPAN) / NAK_SPAN
first_balance = full_years[moon_lord] * (1 - frac_elapsed)
# then walk the fixed lord sequence, sub-dividing each into antardashas

Step 5 — Memory and the product layer

A session pins the chart in the system prompt so follow-ups don't degrade, keeps a rolling window of turns, and meters free questions. This is the ordinary web-app scaffolding — auth, billing, saved readings — that turns a script into a product.

04Making it usable

Three additions moved it from demo to something a real person can use.

Place search with a correct timezone. A Google Places autocomplete on the birth-place field fills latitude and longitude. But the subtle part is the timezone: birth charts need the offset that applied on the birth date, including historical base-offset changes and DST — not today's offset. Rather than a live timezone API, the offset is derived from the coordinates and the birth datetime against the IANA tz database, which encodes those historical rules.

from timezonefinder import TimezoneFinder
from zoneinfo import ZoneInfo

tz   = TimezoneFinder().timezone_at(lat=lat, lng=lon)
off  = datetime(y, m, d, hh, mi, tzinfo=ZoneInfo(tz)).utcoffset()
# Delhi 1990 -> +5.5 · London Jul 1970 -> +1 (BST), handled automatically
astro/geo.py — the ascendant is birth-time sensitive, so this is not a cosmetic detail.

Remembering the last five entries. Recent birth-detail sets are cached client-side and shown as clickable chips, so re-checking a chart is one tap. The server mirrors this with a small LRU cache on the natal computation — the expensive, date-independent part.

Bring-your-own key. An optional key field lets the app run on a user's own credentials, stored only in their browser, with the server environment variable as the default. Convenient for a personal tool; for production you'd keep keys server-side.

05The model is the last step — and it's swappable

Because the chart is computed deterministically, the language model is genuinely pluggable. That reframes the most common question about cost: if I use a cheaper or free model, will the results still be accurate? The answer splits cleanly.

  • Chart accuracy is untouched. Positions, houses, dasha dates — all pure Python. Every model sees the identical, correct chart.
  • Interpretation quality is the only variable. That's a function of the model's reasoning, its fidelity to the rule scaffold, and its non-English output — not of which vendor serves it.

So the abstraction is a single function with a provider registry. Adding a provider is a few lines; switching the default is one environment variable.

def complete(system, messages, *, provider=None, model=None, api_key=None):
    provider = provider or DEFAULT_PROVIDER            # LLM_PROVIDER env var
    if provider == "anthropic":
        return anthropic_call(system, messages, model, api_key)
    # Groq & other OpenAI-compatible endpoints via base_url
    return openai_compatible_call(system, messages, model, api_key)
llm.py — the only part of the pipeline that knows or cares which model answers.

"Groq" isn't a model. It's an inference provider that runs open-source models — so "use Groq instead of Claude" means "use Llama 3.3 70B instead of Claude." For grounded interpretation, that gap is smaller than it sounds.

06The free-tier reality check

A free, fast model is attractive, but the binding constraint usually isn't quality — it's rate limits, and specifically tokens-per-minute. Because the chart context is pinned on every turn, each request is token-heavy, so a generous requests-per-day number can be misleading.

ConcernPractical reading
Model catalogOpen-source only (Llama, Qwen, GPT-OSS, etc.); fine for interpretation.
Free-tier limitsTens of requests/min and a modest tokens/min ceiling, per organization. Great for prototyping, tight for real traffic.
The real bottleneckTokens/min, not requests/day — a long pinned prompt eats the budget fast. Prompt caching helps.
Model choiceA 70B-class model is the strong free pick; the small fast model is noticeably weaker at synthesis and non-English output.

The design that falls out of this is a hybrid: because the provider is per-session, you can default casual questions to a free, fast model and route a "deep reading" to a frontier model like Claude where the extra reasoning earns its cost.

07Run it locally

The whole thing is a handful of files. Everything deterministic lives under astro/; the model is isolated in llm.py; the API and product logic sit in app.py; the UI is one static file.

kundli_app/
├── app.py              # FastAPI: /api/chart, /api/chat, memory, metering
├── llm.py              # pluggable provider layer (Claude / Groq)
├── prompts.py          # the chart-grounded interpretation prompt
├── requirements.txt
├── astro/
│   ├── ephemeris.py    # Step 1 · Swiss Ephemeris compute
│   ├── chart.py        # Step 2 · structured context serializer
│   ├── dasha.py        # Step 4 · Vimshottari timing
│   ├── transits.py     # Step 4 · current transits
│   └── geo.py          # historical timezone from coordinates
└── static/
    └── index.html      # chat UI + North-Indian kundli chart
# Requires Python 3.10+

1 · Install dependencies

# from the project root
cd kundli_app
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate

pip install -r requirements.txt
# or install the packages directly:
pip install pyswisseph fastapi "uvicorn[standard]" pydantic \
            timezonefinder tzdata anthropic openai

2 · Choose a model provider and set keys

The provider is the only thing you configure. Pick one:

# Option A — Claude (default provider)
export ANTHROPIC_API_KEY=sk-ant-...

# Option B — Groq free tier (open-source models, no card)
export LLM_PROVIDER=groq
export GROQ_API_KEY=gsk_...          # from console.groq.com/keys

# Windows PowerShell:
#   $env:LLM_PROVIDER="groq"; $env:GROQ_API_KEY="gsk_..."

3 · Start the server

uvicorn app:app --reload
# → open http://localhost:8000

Enter birth details, generate the chart, and ask a question. You can also switch the provider, model, or key per session from the UI's Keys & settings panel — no restart needed. Optional overrides: ANTHROPIC_MODEL (default claude-opus-4-8) and GROQ_MODEL (default llama-3.3-70b-versatile).

08What I'd harden next

  • Persistence — sessions are in-memory; move to Redis or Postgres.
  • Auth & billing — real accounts and a payment provider behind the free-question meter.
  • More technique — divisional charts (D9, D10) and classical yogas, all derivable from the same raw positions.
  • Streaming — server-sent events for a snappier chat feel.
  • Key hygiene — server-side keys and referrer-restricted map keys in production.

The lesson generalizes past astrology. Whenever a domain has a hard, computable core — finance, scheduling, physics, law with structured statutes — the winning pattern is the same: compute the ground truth deterministically, hand the model exact structured facts, and ask it only to reason and explain. The model is the interface, not the engine. Get the grounding right and even a free model reads like an expert; get it wrong and the best model in the world still sounds like a horoscope.
